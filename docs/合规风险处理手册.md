QCIQ 功能合规与审核风险应对指南

1. 知识产权 (IP) 规避策略

风险等级：⭐⭐⭐⭐⭐ (极高)
直接使用腾讯官方的素材或名称，在微信体系内几乎 100% 会被驳回或投诉。

1.1 视觉素材 (UI/Assets)

❌ 绝对禁止：

使用经典的企鹅 Logo。

使用“QQ”、“Qzone”、“QQ秀”等官方商标词汇。

直接提取并使用原版软件的图标文件。

✅ 修改建议：

吉祥物：改用像素风的“电子狗”、“复古电脑”或“外星人”作为 Logo。

名称：

QQ -> QCIQ (保留形似但不同) 或 M-Chat (Millennium Chat)。

Qzone -> My Space (我的地盘) 或 Cyber Zone (赛博空间)。

黄钻/红钻 -> 星钻/月钻 或 黄金会员/红宝石会员。

UI 风格：保持“神似而形不似”。使用 Windows 98 的灰色风格，而不是 QQ 2005 标志性的蓝色渐变，强调“操作系统模拟”而非“软件克隆”。

1.2 音效

❌ 禁止：使用原版“嘀嘀嘀嘀嘀”提示音（这也是注册商标的一部分）。

✅ 建议：使用 8-bit 风格的合成提示音，或者通用的机械键盘打字声、老式电话铃声。

2. 小程序类目与资质 (Category Audit)

风险等级：⭐⭐⭐⭐ (高)
个人开发者没有社交类目资质，也通常无法申请“游戏”类目所需的版号。

2.1 类目选择策略

❌ 避雷类目：

社交、论坛、社区 (需企业资质 + ICP)。

游戏 (个人开发者发布游戏虽然微信允许，但特定玩法可能触发生态管控，且无法开通支付；建议避开直接作为“游戏”提审，以免面临复杂的版号或备案审查)。

✅ 推荐类目 (以“工具”名义上架)：

工具 > 娱乐 (最稳妥，涵盖模拟器、生成器、解压玩具)。

工具 > 图片处理 (如果主打头像生成)。

生活服务 > 休闲娱乐。

2.2 功能包装话术 (De-Gamification)

为了让审核员认为这是一个“工具”而非“游戏”，文案需要去游戏化。

❌ 避免词汇：

“游戏大厅”、“副本”、“关卡”、“Boss”、“充值”、“战力”。

✅ 替换词汇：

游戏 -> “互动体验”、“减压模拟”、“禅模式”。

扫雷 -> “逻辑训练器” (Star Explorer)。

农场 -> “电子盆栽” 或 “专注力花园”。

宠物 -> “桌面精灵” 或 “数字伴侣”。

等级 -> “成长记录” 或 “活跃指数”。

提审描述示例：

“本应用是一款怀旧风格的多功能工具箱，包含火星文翻译、复古头像制作、电子宠物模拟养成（用于缓解压力）以及逻辑思维训练模块，旨在通过千禧年视觉风格带给用户沉浸式的怀旧体验。”

3. 特定功能整改 (Feature Specifics)

3.1 漂流瓶 (Drift Bottle)

风险：微信官方曾因色情低俗内容下架了微信漂流瓶。小程序对类似功能审核极严。

✅ 修改方案：

改名：改为**“时空胶囊”或“树洞信号”**。

机制阉割：

完全切断用户间的文本传输。

仅允许用户选择预设的 Tag（如“想要暴富”、“求好运”）发送。

或者，捞到的内容100% 为 AI 生成的趣味段子/鸡汤，不显示其他真人的内容。

底线：必须让审核员确信，这是一个“人机交互”功能，而不是“人人交互”。

3.2 家族/群聊 (Clans)

风险：被视为“群聊”或“社区”，触犯 UGC 监管底线。

✅ 修改方案：

去社交化：将“加入家族”变成一种**“佩戴勋章”**的行为。

无聊天室：家族页面仅显示“家族宣言”和“排行榜”，不提供家族成员之间的聊天/发帖功能。

排行逻辑：仅比拼“活跃度数值”，不比拼“内容”。

3.3 虚拟支付 (iOS Payment)

风险：小程序在 iOS 端禁止购买虚拟物品（如充值黄钻、买宠物饲料）。

✅ 修改方案：

安卓端：可以接入微信支付（需企业资质）。个人开发者通常无法接入支付。

通用方案 (推荐)：完全依赖广告变现。

开通黄钻 = 观看 3 次激励视频。

购买饲料 = 观看 1 次激励视频。

避坑：界面上不要出现“¥”、“元”、“充值”字样，统称为“兑换”或“解锁”。

3.4 排行榜 (Leaderboard)

风险：展示用户列表可能被判定为“诱导社交”或“泄露隐私”。

✅ 修改方案：

不可点击：榜单上的头像和昵称禁止点击，点击后不能跳转到该用户的个人主页，也不能弹出“加好友”选项。

匿名处理：如果担心隐私，可以将昵称中间打码（如 葬爱**少），或者仅展示通过审核的 QCIQ 昵称（而非微信真实昵称）。

数据脱敏：仅展示虚拟数据（等级、宠物心情），不展示真实地理位置等敏感信息。

3.5 空间与互动风险 (Space & Interactions)

风险：留言板（Guestbook）若允许访客输入文字，将被直接判定为“社区/论坛”功能。

✅ 修改方案：

禁止自由留言：访客在别人的空间里，没有输入框，只有一个“踩一下”的按钮。

系统化留言：点击“踩一下”后，系统自动在留言板插入一条记录：“

$$访客昵称$$

 来踩过你了，人气 +1”。

AI 氛围组：为了不让留言板空荡荡，可以混入系统自动生成的 AI 留言。

访问限制：不要做一个“随便逛逛”的广场功能。用户只能通过微信分享卡片点进特定好友的空间。

关于“回踩”与头像跳转：

✅ 合规逻辑：点击访客记录中的头像 -> 直接跳转到对方的空间页面（Game Scene） -> 进行“踩/偷菜”操作。

❌ 违规逻辑：点击头像 -> 跳转到“个人资料卡” -> 显示“加好友”、“发消息”按钮。

核心区别：只要点击头像后的行为是**“去对方家里玩游戏”（类似支付宝蚂蚁森林去朋友家收能量），而非“查看对方联系方式”**，这就属于游戏逻辑，是安全的。

4. 内容安全 (Content Safety) 实操清单

即便规避了社交属性，用户上传的头像和签名依然属于 UGC，必须接审核。

必须接入的 API：

security.msgSecCheck (文本检测)：用于昵称、签名、宠物命名。

security.mediaCheckAsync (图片检测)：用于自定义头像上传。

前端表现：

用户修改资料后，不要立即更新。

显示“审核中...”状态。

等待云函数回调返回 Pass 后，再更新数据库和 UI。

兜底机制：

如果接口超时或失败，默认视为“不通过”或维持原样，宁可错杀不可放过。

5. 总结：如何过审？

核心口径：

“这是一个单机怀旧模拟器，所有的‘好友’都是代码写死的 NPC，所有的‘互动’都是基于规则生成的。这不仅不是一个社交软件，甚至连联网功能都只是为了更新游戏资源。”

只要坚持**“看似社交，实则单机/人机”**的原则，就能在最大程度上规避版权和资质风险。
